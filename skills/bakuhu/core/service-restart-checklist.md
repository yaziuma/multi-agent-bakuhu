# サービス再起動チェックリスト（Service Restart Checklist）

## 絶対ルール

**サービスを起動・再起動する前に、必ず管理スクリプトの存在を確認せよ。**

確認なしの手動起動は禁止。違反は殿の逆鱗に触れる。

## なぜこのルールが必要か

- プロジェクトには起動手順・環境変数・引数が必要な場合がある
- 手動起動で設定を漏らすと、サービスが正常動作しない
- 既存スクリプトは過去の失敗から学んだ知恵が詰まっている。無視するな

## 確認手順

```
Step 1: プロジェクトの管理スクリプトを探す（下記の優先順位で確認）
Step 2: スクリプトが見つかった場合 → そのスクリプトを使用して起動・再起動
Step 3: スクリプトが見つからない場合 → 手動起動を実施し、理由を報告に明記
```

## スクリプト確認の優先順位

以下の順番で確認せよ（上位が見つかれば下位は不要）:

| 優先順位 | 確認対象 | 確認コマンド例 |
|----------|----------|----------------|
| 1 | `scripts/*.sh`（start_server.sh, stop_server.sh, restart.sh 等） | `ls scripts/*.sh` |
| 2 | `Makefile`（`make start`, `make restart` 等） | `cat Makefile \| grep -E '^(start\|stop\|restart)'` |
| 3 | `docker-compose.yml` / `compose.yaml` | `ls docker-compose.yml compose.yaml 2>/dev/null` |
| 4 | `systemd` unit ファイル | `ls /etc/systemd/system/*.service 2>/dev/null` |
| 5 | プロジェクトルートの `README.md`（起動手順記載の可能性） | `grep -i 'start\|run\|launch' README.md` |

## スクリプトが見つかった場合

- そのスクリプトを**そのまま**使用せよ
- スクリプトの内容を変更するな（タスクに明示的な指示がない限り）
- 例: `bash scripts/start_server.sh`, `bash scripts/restart.sh`

## スクリプトが見つからない場合

- 手動起動を実施してよい
- ただし、報告書に以下を必ず記載せよ:
  1. どのスクリプトを探したか（確認したパス一覧）
  2. 見つからなかった理由（ファイルが存在しない等）
  3. 代わりに使用した手動起動コマンド

## 禁止事項

| 禁止行為 | 理由 |
|----------|------|
| スクリプト確認なしの即時手動起動 | 管理スクリプトを無視する最悪の失敗パターン |
| スクリプトがあるのに手動起動 | 環境変数・引数・依存関係を漏らすリスク |
| 手動起動時に理由を報告しない | 次回以降の再現性が失われる |
| `sudo` / `kill -9` の無断使用 | 破壊的操作禁止（D005/D006 参照） |

## 実行チェックリスト

サービス起動・再起動の前に確認:

- [ ] `scripts/*.sh` を確認したか
- [ ] `Makefile` を確認したか
- [ ] `docker-compose.yml` を確認したか
- [ ] `systemd` unit を確認したか
- [ ] スクリプトが見つかった場合、それを使うか
- [ ] スクリプトがない場合、理由を報告に記載するか

## 適用範囲

- 全エージェント（shogun / karo / ashigaru / gunshi / bugyo 等）
- 全プロジェクト（bakuhu, CineBox, tenshukaku, queue-viewer, portal 等）
- **「急いでいるから」は免罪符にならない。確認してから起動せよ。**
