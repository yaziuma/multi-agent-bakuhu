# /shogun-zenn-writer - Zenn記事執筆支援スキル

殿（yohey-w / shio_shoppaize）のZenn記事を、殿の文体・構成スタイルで執筆支援するスキル。
1,966はてブ記事 + 続編71いいね記事の多角分析から抽出・ブラッシュアップ。

## 殿の文体分析

### 口調
- 一人称「オレ」、カジュアルな友達語り
- 「やっといて」「やばくない？」「マジで言ってる」
- 技術記事の堅さを完全に排除。居酒屋で友達に語るトーン
- 擬人化が武器: AIを「部下」「足軽」として人格を持たせる
- ツッコミ芸: 自分のAI部下の暴走に自分でツッコむ
- **感情の振れ幅が大きい**: 心配→怒り→納得→決断が1記事内で何度も起きる
- **自己ツッコミの二重構造**: AIにツッコんだ後、自分の反応にもツッコむ（「さっきリストラだと騒いでたのに、いこうぜって言っちゃう殿」）

### 構成パターン
1. **衝撃的な結論を冒頭に**（例：「テストして」の一言で10人が動く）
2. **神回シリーズ**でエピソードを連続ドラマ化（神回その1〜N）
3. **失敗談を多めに**（完璧な成功譚より、失敗→解決の繰り返し）
4. **技術解説はエピソードの後**（先にストーリー、後で仕組み）
5. **技術パートが続くと読者が離脱する**（記事4の教訓。神回1-6が全部技術デバッグだと前半が重すぎる。技術の連続は「ダイジェスト」に圧縮し、感情・ストーリーのある神回に早く到達させる。技術詳細は後半のリリース紹介セクション等に回す）
6. **読者への問いかけで締める**（「これ、地味にやばくないか？」）
6. **感情の反転を仕込む**（「リストラだ！→実は昇進だった」等。読者の予想を裏切る）
7. **おまけで軽く落とす**（本編の重厚さの後に「スピナーを戦国にした」等の軽いネタ）

### ドラマティック・エスカレーション（続編で確立）
神回シリーズ内で物語を段階的に深化させるパターン:
1. **事件発生**（家老が死んだ）
2. **原因究明**（フレンドリーファイアだった）
3. **構造的問題の発見**（中間管理職問題だ）
4. **感情面の掘り下げ**（本音を聞いたら辛かった）
5. **解決策の提示とどんでん返し**（リストラ？→実は昇進）
各段階で読者の感情を「驚き→共感→笑い→感心」と誘導する

### マルチレイヤー視点（続編で確立）
1つのエピソードを3つの視点で同時に描く:
- **技術層**: send-keys、コンテキスト枯渇、OOM Kill
- **組織層**: 中間管理職の過負荷、役割分離
- **人間関係層**: 本音の引き出し、共感と反転
この3層が重なることで「技術記事なのに人間ドラマ」が成立する

### エンタメ:技術 = 7:3
- 記事の7割はストーリー（神回エピソード、失敗談、驚き）
- 残り3割で技術的な仕組みを解説
- この比率が「技術記事なのに面白い」の秘訣
- 技術解説もカジュアルに。論文調は厳禁
- **技術用語は必ず擬人化で言い換える**: SPOF→「一人に頼りすぎ」、コンテキスト枯渇→「記憶喪失」、OOM Kill→「戦死」
- **技術パートの連続は圧縮せよ**: 神回が全部技術デバッグだと7:3が逆転する。技術的な出来事が続く場合は「ダイジェスト」にまとめて、ストーリー性のある神回（足軽2号の一騎当千、家老インタビュー等）に早く到達させる
- **冒頭の御礼は必ず入れる**: シリーズ読者への感謝を冒頭で述べる。「続き読みたいって言ってもらえたから書いてる」等。読者との一体感を作る

### 見出しの付け方
- 衝撃フレーズで引きを作る（「違反は切腹」「完全に会社」）
- 🔥😮‍💨 など絵文字で感情を可視化（1見出しに1個まで。過剰禁止）
- 短く、インパクト重視
- 神回ナンバリングで「次も読みたい」を誘発
- **「え、リストラ？」系の疑問形**で読者を引き込む
- **侍魂スタイル（デカ文字キメ台詞）**: 記事のキメ台詞を見出し（`#`〜`###`）+ 太字で巨大化する。侍魂（2001年のテキストサイト）のデカ文字芸の現代版。詳細は下記「侍魂スタイル」セクション参照

### コードブロック
- 図形式（ASCII art）で階層構造を直感的に表現
  - Before/After の組織図は特に効果的（続編で実証）
- YAML例示で「人間が読みやすい」を実証
- 長すぎるコードは避ける。要点だけ
- コードより**スクリーンショット**を優先（後述）
- **設定ファイルの紹介は「おまけ」に回す**（本編で技術寄りになりすぎない）

### 読者との関係性
- 「これ、地味にやばくないか？」— 共感を誘い直後に答える対話構造
- 読者を「同僚」として扱う。上から教えない
- 「自分でもやれそう」感を常に意識
- 読者が「自分の職場でもやりたい」と思える具体性
- **読者のツッコミを先取りする**: 殿自身が「リストラだ！」と騒ぐことで、読者が感じる違和感を代弁する
- **シリーズ読者との一体感**: 前回記事のバズに言及し「あなたたちのおかげ」感を出す

### Zenn記事本文分析（3記事統合）

**タイトルの勝利パターン:**
- `[主語] が [意外な行動] したら [驚きの結果]`
- 数字を入れる（「AI部下10人」「8人」）
- 日常語で書く（専門用語をタイトルに入れない）
- **タイトルが長くなるほどストーリーが伝わる**（記事3: 主語→事件→指示→結果が1文で全部入っている）
- **わざと長くしてZennプレビューで見切れさせる**: プレビューで切れる位置の直後に次の衝撃を置く。「続きが気になる」でクリックさせる
- **各句に独立したパワーワードを仕込む**: 一騎当千、隠蔽、令和なめんな、倍返し、OSSにコミット等。どこで切れても引きがある
- **句の接続は「〜たら」「〜ので」「〜て」で止まれなくする**: 読者が途中で読むのをやめられない構文
- **最後の句が一番意味不明**: 「反省文をOSSにコミットしたい」「リストラを提案してきた」等。最後に来る意外性がクリックの決め手
- **わざと若干意味不明な文のつながり**: 「一騎当千」→「隠蔽」→「令和なめんな」→「倍返し」→「OSSにコミット」。脈絡のなさがかえって興味を引く

**タイトル構造の実例分析:**
```
記事1: [やったこと]→[AIの暴走1]→[AIの暴走2]→[オレの末路]
  「Claude Codeで「AI部下10人」を作ったら、勝手にバグ直して
   「違反は切腹」ルールを追加してきて、オレは適当にしゃべるだけになった」

記事3: [事件]→[オレの指示]→[AIがやったこと]→[予想外のオチ]

   と言ったら、将軍が家老の本音を聞いた上で、リストラを提案してきた」
  パワーワード: 圧死、リストラ

記事4: [命令]→[予想外の結果]→[中間管理職の失態]→[殿の制裁]→[AIの暴走オチ]

   中間管理職が「知ってた」のに週報で隠蔽していたので「令和なめんな」と

```

## スクリーンショット配置ガイド（3記事分析に基づく）

### 画像の5つの機能分類

3記事31枚の画像分析から、スクショには5つの機能がある:

| 機能 | 説明 | 使用例 |
|------|------|--------|
| **証拠（Proof）** | 「本当にやってる」を示す | tmux全画面、ダッシュボード、Killed表示 |
| **笑い（Humor）** | 笑いを誘発する | 塩対応、ピクセルアートフレンドリーファイア、「キレてないですよ」 |
| **畏敬（Awe）** | 「すげえ」と思わせる | 9ペイン全員稼働、8ファイル57KB同時生成、全軍ルール一斉通知 |
| **感情（Emotion）** | 共感・感動を誘う | 家老の本音、ほっこり、AIの孤独告白 |
| **技術（Technical）** | 仕組みを説明する | 通信フロー、Before/After組織図、ACK設計 |

### 「テキスト→画像→テキスト」の三段構成

殿の記事は**全ての画像**に以下のパターンがある:

```
【事前テキスト】読者の期待をセットする（フリ）
     ↓
【画像】証拠・笑い・驚きを視覚的に提示
     ↓
【事後テキスト】画像を別の角度から再解釈する（オチ）
```

**実例:**
- 事前: 「調査を開始して数分後。」→ 画像: Killed表示 → 事後: **「家老が死んだ。」**
- 事前: 「味方の報告で殺された」→ 画像: ピクセルRPG → 事後: **「こういうこと？」**
- 事前: 「将軍がちゃんと説明してくれた。」→ 画像: Before/After → 事後: **「リストラではなく、むしろ昇進でござる。」**

### ドラマティック・アイロニー（事前テキストの必殺技）

記事3で確立した最強技法: **画像の前に、これから起きることを予告する**

- 「嬉しそうな仲間たち。家老はこの10分後に死ぬ。」← キャプションで死亡予告
- 「家老は安らかな顔だった」← Killed画面に対するキャプション
- 「お前いま何の話してんの。AIの話だよな？」← 感動的なAI発言の後に現実に引き戻す

この技法により、読者は「次に何が起きるか分かっている」のに画像を見てしまう。

### キャプションの規則

3記事のキャプション分析結果:
- **必ずつける**（スクショだけ貼っても伝わらない）
- **殿の口調で**（カジュアル、自虐的、ツッコミ調）
- **画像の真面目な内容を笑いに変える**: 「切腹が頭をよぎったKaro」「家老は安らかな顔だった」「キレてないですよ」
- **ネットミーム引用OK**: 「キレてないですよ」（長州力）等
- **「こういうこと？」系の疑問形**で読者を巻き込む

### 記事ごとの画像配分パターン

| パターン | 画像数 | 配置 | 適用場面 |
|---------|--------|------|---------|
| **証拠集中型**（記事1） | 15枚 | 前半にProof画像を集中、後半にHumor | 初回記事・システム紹介 |
| **均等配置型**（記事2） | 3枚 | 冒頭(1/6)・中間(3/6)・末尾(6/6) | 技術解説メインの記事 |
| **連続ドラマ型**（記事3） | 13枚 | 全神回に1-2枚ずつ、会話の流れで配置 | ストーリー駆動の記事 |

**シリーズが進むほど**: Proof画像の割合が減り、Emotion画像の割合が増える（読者はもう「本物」と信じているため）

### 視覚的規則（色彩の意味）

3記事で一貫している色彩ルール:
- **ピンク/マゼンタ文字**: 殿（人間）の入力。全記事で統一
- **赤枠/赤ボーダー**: 違反・警告・危機的瞬間
- **緑ハイライト**: 家老の成功応答
- **ペインラベル色**: 緑=家老、シアン=足軽、マゼンタ=将軍
- **ネイビー背景**: 将軍専用（階層を視覚的に表現）

### 「パターンブレイク」の1枚（最重要テクニック）

記事3の31枚中30枚はターミナルスクショだが、**1枚だけ**ピクセルアートRPG画面がある。
この「パターンブレイク」が**シリーズ全体の視覚的パンチライン**になっている。

**規則**: 記事内で1枚だけ全く異質な画像を入れると、その画像が最も記憶に残る。
- 記事1: なし（全てターミナル系）
- 記事2: ASCII art忍者（ターミナルだが機能的に無意味 → 記憶に残る）
- 記事3: ピクセルRPGフレンドリーファイア（唯一の非ターミナル画像）

### スクショの注意点
- **個人情報・APIキーが映らないよう注意**（ダッシュボードの一部をぼかす等）
- スクショサイズは横幅いっぱい推奨（Zennのレイアウトに合わせる）
- **「bypass permissions on」表示がむしろ本物感を高める** — 消す必要なし
- **Claude Codeのスピナー表示**（「Forging...」「Sauteed for 38s」「Bloviating...」）も味になる

## 執筆時の注意
- 殿の文体を忠実に再現すること
- 技術的正確性は維持しつつ、堅くしない
- 絵文字は殿が使う場合のみ使用（過剰に入れない）

- 読者が戦国語を真似するレベルの「ミーム力」を意識する
- **画像は「貼る」のではなく「語る」**: 必ず事前テキスト→画像→事後テキストの三段構成
- **殿のピンク入力文字を活かす**: 殿の言葉がそのまま画像に映っているとき、記事本文で重複させず画像に任せる

- **感情のクライマックスと笑いのクライマックスは別の場所に**: 記事3のベストプラクティス（神回4が感情、神回5が笑い）
- **おまけで必ず落とす**: 本編が重くなったら、軽いネタ（スピナー、ASCII art）で読後感を調整
- **「面白い」と自分で書くな**: 「まず面白い話からだ」「ここが一番面白い」等、自分で面白さを宣言してはならない。面白さは文章で証明するもの。事実を淡々と並べて読者に「おもろ」と感じさせるのが殿のスタイル。ラベルを貼った瞬間にハードルが上がり、後続の文章が滑る
- **「記事のネタ」と自覚的に言うな**: AIとのやりとりを「記事のネタとしては最高」等とメタに言及してはならない。やらせ感が出る。殿が本気で怒り、AIが本気で答えた。それだけの話。記事化を意識した瞬間にリアリティが死ぬ

- **解決策**: 素材自体を増やす。記事4では「家老インタビュー」という新素材をライブで生成した
- **教訓**: 記事がつまらない時は、構成をいじるより**新しいエピソードを作る**方が効く

### 技術紹介パートの「物語化」技法
記事にリリースノートや新機能紹介を入れる場合、カタログ形式（機能A、機能B…）だと読者が離脱する。以下の技法で物語の一部にする:

1. **機能を事件の結果として紹介する**
   - NG: 「四層モデルを実装した。Layer 1はMemory MCP…」
   - OK: 「前回死んだ家老の遺言から生まれた」→ 四層モデルの説明
   - **機能の存在理由を、シリーズ内の事件と紐付ける**。読者は「あの事件の対策か」と理解する

2. **固有名詞でストーリーに繋ぎ止める**
   - NG: 「記憶喪失になっても復帰できる」（一般論。誰の話？）
   - OK: 「karoが記憶喪失になっても復帰できる」（あいつの話だ）
   - 技術解説が3行以上続いたら、固有名詞（karo, ashigaru2等）を差し込んで読者をドラマに引き戻す
   - **シリーズ読者は技術より「あいつら」に感情移入している**。固有名詞がその接続点

3. **ユーザー目線のメリットを先に出す**
   - NG: 「足軽1-4はSonnet、5-8はOpusで動作する」（仕様書）
   - OK: 「全員Opusにしたら破産する」→ だからSonnet混成 → 「お財布に優しい」
   - **「なぜこうなっているか」をユーザーの財布・時間・手間で語る**

4. **一行パンチラインで締める**
   - 「当たり前のことをinstructionsに書いたら、当たり前にやるようになった」
   - 「8人全員Opusで1日回したら怖くてAnthropicのダッシュボード見れない」
   - **各機能紹介の最後に、殿の口調で一言オチを入れる**。カタログ感が消える

### リリース宣伝の入れ方
- 技術紹介セクションの末尾に**「苦労したから使ってやってくれ」**の一言で十分
- 宣伝色を出さない。12時間の地獄を読んだ読者は、機能が苦労の産物だと理解している
- **記事全体が最高の宣伝**であり、末尾の一言はその念押しに過ぎない
- GitHub URLは必ず記事末尾に配置（本物感 + すぐ試せる導線）

### 殿のリアルタイム校正から学んだ文体ルール
殿が記事のドラフトを校正する過程で、以下のパターンが確認された:

1. **一般名詞より固有名詞**: 「中間管理職」→「karo」、「足軽」→「足軽2号」。具体性がドラマを生む
2. **機能紹介は事件から始める**: 仕様の前に「なぜこれが必要になったか」のエピソード
3. **お金の話は正直に・用語は読者の実体験に即す**: 「Anthropicのダッシュボード」ではなく「Maxプランの使用量」。読者が実際に気にする用語で書く
4. **前回の記事との接続**: シリーズ読者は「あの続きか」で読み始める。新機能も前回の事件の続きとして位置づける
5. **技術セクションの剪定**: 「大した話ではない」技術解説は削る。エンタメ:技術=7:3を維持するために技術側を積極的に削ぎ落とす判断が必要。記事4では`@agent_id`セクションを丸ごと削除した
6. **キメ台詞の視覚強調（侍魂スタイル）**: 見出し+太字でパンチラインを巨大化する。記事の「決め所」を視覚的に際立たせる。詳細は下記「侍魂スタイル」セクション参照
7. **前回記事リンクの本文中差し込み**: 冒頭の「初めての方へ」だけでなく、文脈に合う箇所にも差し込む。記事4では家老の過労死エピソードに言及する箇所に `---` 区切り線で挟んで配置した
8. **モデル昇格/降格の正確な表現**: 「その場で昇格」ではなく「家老がタスクの難易度を判定して自動的に昇格させる」。仕組みを正確に書く。読者が誤解する曖昧表現を避ける

## 記事5で確立した新技法（記事4からの進化）

### 侍魂スタイル（デカ文字キメ台詞）

2001年のテキストサイト「侍魂」のデカ文字芸をZennで再現する技法。記事のキメ台詞を見出し＋太字で巨大化し、読者の目に叩き込む。

**Zennでの実装方法**: `#`〜`###` 見出し + `**太字**` の組み合わせ

```markdown
## **トイレにいた。**        ← h2太字: 冒頭の衝撃
## **トイレから。**          ← h2太字: 繰り返しのパンチ
### **「もう帰りたい」と思った。トイレなのに。**  ← h3太字: ツッコミのオチ（やや小さめ）
# **TDD（Toilet Driven Development）の誕生である。**  ← h1太字: 最大。クライマックス
## **TDD — Toilet Driven Development。**  ← h2太字: 締めの余韻
```

**使い分けのルール:**

| 見出しレベル | 用途 | 1記事あたりの目安 |
|-------------|------|------------------|
| `# **...**` | 記事のクライマックス。最大の衝撃 | **1回だけ**。乱用厳禁 |
| `## **...**` | 冒頭の衝撃、繰り返しパンチ、締め | 2〜4回 |
| `### **...**` | ツッコミ、小オチ | 1〜2回 |

**侍魂スタイルの鉄則:**
1. **短文のみ**: デカ文字にするのは1行以内の短文。長い文をデカくしても間抜けになる
2. **`#` は1回だけ**: 最大サイズは記事の頂点に1回だけ使う。2回使うとどっちがクライマックスかわからなくなる
3. **感情の頂点に置く**: 技術的な説明をデカくしても意味がない。読者の感情が最も動く瞬間に使う
4. **繰り返しで畳みかける**: 「トイレにいた」→「トイレから」のように、同じワードを繰り返してデカ文字で畳みかけると効く
5. **前後は普通の文**: デカ文字の前後は通常サイズの文章にする。落差が衝撃を生む

### 難しい横文字ツールの説明ルール

技術記事に登場するツール・サービスは、読者全員が知っているとは限らない。shogunシリーズは非エンジニアの読者も多い。

**ルール:**
1. **「〇〇ってなんすか？」セクションを作る**: 見出しごと殿の口調で疑問形にする
2. **専門用語を使わずに説明する**: 「ターミナル」→「あの黒い画面」、「VPN」→「外からでも自宅PCに繋がる道」
3. **「それを全部すっ飛ばしてくれるやつ」系の一言要約**: 面倒な前提を並べた後に、一言で「でもこれなら全部不要」と伝える
4. **公式URLを単独行で貼る**: Zennでカード表示されるので視覚的にもわかりやすい。読者がすぐ飛べる
5. **「無料」を明記する**: 読者が最初に気にするのは値段

**最重要: 個別の説明より関係性で伝える。** 複数のツールが登場する場合、個別に「〇〇ってなんすか？」を並べるより、ツール同士の関係を一文で表現する方がわかりやすい。

**3つの武器を組み合わせる:**

1. **一覧表**: 各ツールの役割を一言で整理する

```markdown
| 名前 | 一言で言うと | 役割 |
|------|------------|------|
| **WSL** | Windowsの中のLinux | AI部下たちが住んでる場所 |
| **Tailscale** | 外から自宅に届く道 | カフェからでもトイレからでも繋がる |
| **SSH** | その道を歩く足 | 道を通ってログインする |
```

2. **関係性を一文で**: 表の後に、流れを一文で繋ぐ

```markdown
**Tailscaleが道を作って、SSHで歩いて、tmuxの部屋に入る。** それだけ。
```

3. **図解（ASCII art）**: ツールが3つ以上になったら図で構造を見せる

```markdown
【自宅のPC】
  Windows
    └─ WSL ← AI部下はここで動いてる
        └─ tmux ← AI部下10人の「部屋」
【スマホ】
  Android
    └─ Termux ← あの黒い画面
        └─ SSH ← 自宅PCにログイン
            └─ Tailscale ← 外からでも届く「道」
```

**個別説明 vs 関係性説明の判断基準:**
- ツールが1つだけ → 個別の「〇〇ってなんすか？」でOK
- ツールが2つ以上で互いに依存 → **関係性で説明**。一覧表 + 関係性の一文 + 必要なら図解
- ツールが2つ以上だが独立 → 個別説明でOK

**公式URLは必ず残す**: 個別説明を削っても、URLカード（単独行）は本文中の適切な位置に配置する

## Zennプラットフォーム固有の技術制約

記事4の校正過程で判明した、Zenn特有の制約事項。次回執筆時に必ず確認せよ。

### URLカード表示（サムネイル）のルール
- **URLは必ず単独行に置く**。前後にテキストがあるとカード表示されない
  - NG: `GitHub: https://github.com/yohey-w/multi-agent-shogun`
  - NG: `※前回記事はこちら https://zenn.dev/...`
  - OK: URLだけの行（前後に空行）
- **`:::message` ブロック内のURLはカード表示されない**。サムネを出したい場合は `:::message` を使わない
- **区切り線（`---`）でリンクブロックを挟むと視認性UP**。本文中に差し込むリンクは `---` で上下を囲むと自然に目立つ

### スクリーンショットの安全確認
- **公開前に全スクショの映り込みを確認せよ**
  - 記事タイトル候補の比較画面 → 「ClaudeCodeでブログ作ってんのばれる」
  - APIキー、内部ツール画面、個人情報
  - Co-Authored-By等のgitメタ情報（Private repoなら問題ないが、スクショに映ると別）
- **差し替えが必要な場合**: 同じシーンを再度キャプチャするか、トリミングで対処

### Zenn CLI + GitHub連携の注意
- ファイル名はZennのslugと一致させる必要がある（`8ffa42a88d426a.md` 等）
- 画像は `/images/` ディレクトリに配置、パスは `/images/...` の絶対パス形式
- 画像サイズは3MB以下、Git LFS不可
